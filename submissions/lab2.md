### 2.4 — Logs analysis

Generated traffic (`GET /events` + `POST /events/1/reserve`) and traced it through gateway → events by matching timestamps:


### 2.5 — Network inspection

```
$ docker network ls | grep app
ce83145908dd   app_default   bridge   local

$ docker network inspect app_default --format '{{range .Containers}}{{.Name}}: {{.IPv4Address}}{{"\n"}}{{end}}'
app-redis-1:    172.20.0.4/16
app-postgres-1: 172.20.0.3/16
app-gateway-1:  172.20.0.6/16
app-payments-1: 172.20.0.2/16
app-events-1:   172.20.0.5/16
```

### Answer — How does the gateway find `events`?

Docker Compose creates a bridge network (`app_default`) with an embedded DNS server at `127.0.0.11` (visible in `/etc/resolv.conf` inside every container). When the gateway code calls `http://events:8081`, the container's libc resolver queries `127.0.0.11`, which resolves the service name `events` to the `app-events-1` container's current IP — **`172.20.0.5`**. No hardcoded IPs anywhere in config; if the `events` container is recreated and gets a new IP, Docker's DNS just returns the new address.

It's still good practice (prevents accidentally baking in `.env`/`.git` if they're ever added to these directories).

### 2.8 — Non-root user

Added to each Dockerfile before `CMD`:

```dockerfile
RUN addgroup --system app && adduser --system --ingroup app app
USER app
```

No `chown` needed — none of the services write to `/app` at runtime.

```
$ docker exec app-gateway-1 whoami
app
$ docker exec app-events-1 whoami
app
$ docker exec app-payments-1 whoami
app
```

All 3 containers now run as `app` (not root), and `docker compose ps` / `curl http://localhost:3080/health` confirm everything still starts and works correctly.

## Bonus Task — Trace a Request Across Services

### Reserve (`POST /events/1/reserve`)

| Timestamp (UTC) | Service | Action |
|---|---|---|
| 20:09:04.825693 | **events** | `Reserved 1 tickets for event 1: 045f575c-...` (DB + Redis write) |
| 20:09:04.826429 | **events** | responds `POST /events/1/reserve` → 200 OK |
| 20:09:04.827306 | **gateway** | receives events' 200 OK (httpx log) |
| 20:09:04.828281 | **gateway** | responds `POST /events HTTP/1.1` → 200 OK to client |

→ gateway → events round trip ≈ **2.6ms** (`.825693` → `.828281`).

### Pay (`POST /reserve/{id}/pay`) — charge → confirm

| Timestamp (UTC) | Service | Action |
|---|---|---|
| 20:09:04.859302 | **payments** | `Payment success: PAY-562414CE for 045f575c-...` |
| 20:09:04.859813 | **payments** | responds `POST /charge` → 200 OK |
| 20:09:04.860516 | **gateway** | receives payments' 200 OK (httpx log) |
| 20:09:04.865854 | **events** | `Order confirmed: 045f575c-...` (writes order row, clears Redis reservation) |
| 20:09:04.866296 | **events** | responds `POST /reservations/{id}/confirm` → 200 OK |
| 20:09:04.866697 | **gateway** | receives events' confirm 200 OK (httpx log) |
| 20:09:04.867388 | **gateway** | responds `POST /reserve/{id}/pay HTTP/1.1` → 200 OK to client |

→ gateway → payments (charge) hop ≈ **1.2ms** (`.859302` → `.860516`).
→ gateway → events (confirm) hop ≈ **0.8ms** (`.865854` → `.866697`).
→ gap between charge-done and confirm-start ≈ **5.3ms** (`.860516` → `.865854`) — gateway issuing the second outbound call (events confirm).

### Answer — End-to-end time

The `/pay` request's full chain — gateway → payments (charge) → gateway → events (confirm) → gateway → client — spans **`20:09:04.859302` to `20:09:04.867388` ≈ 8.1ms** total. All three services run on the same host over the Docker bridge network, so the time is dominated by application processing (JSON serialization, DB/Redis writes), not network latency.
